# Task: State Management

## Description

Implement the Zustand store with persist middleware for build state management. This includes 5 independent build slots, skill unlock/lock actions with prerequisite enforcement, derived stat computation, decision logging, and Zod schema validation for localStorage integrity. This store becomes the single source of truth that both the tree (Task 3/4) and sidebar (Task 6) read from.

## Acceptance Criteria

- [ ] `src/stores/build-store.ts` exports a Zustand store with the following state:
  - `activeSlot: number` (0-4)
  - `builds: PlayerBuild[]` (5 slots)
  - `hoveredSkillId: string | null`
- [ ] Store actions:
  - `setActiveSlot(index)` — switches active build
  - `renameBuild(index, name)` — updates build name
  - `unlockSkill(skillId)` — validates prerequisites, adds to unlocked set, appends to decision log, updates stats
  - `lockSkill(skillId)` — validates no dependents unlocked, removes from set, removes from log, updates stats
  - `resetBuild(index)` — clears to base skills only
  - `setHoveredSkill(skillId | null)` — tooltip state
- [ ] Each `PlayerBuild` initialized with:
  - `unlockedSkillIds`: base skill IDs
  - `decisionLog`: empty
  - `stats`: `{ damage: 0, movement: 0, reduction: 0 }`
  - `name`: `"Build 1"` through `"Build 5"`
- [ ] Stat computation: sum `statBonusAmount` across all unlocked skills, grouped by `statBonus` type
- [ ] `persist` middleware saves to localStorage under key `"skill-tree-builds"`
- [ ] `src/lib/storage.ts` defines Zod schemas for `PlayerBuild` and validates on hydration
- [ ] Corrupt localStorage gracefully resets to defaults (no crash, no blank screen)
- [ ] State survives page refresh
- [ ] TypeScript compiles without errors

## Technical Details

### Zustand Store Structure
```typescript
import { create } from "zustand";
import { persist } from "zustand/middleware";

interface BuildState {
  activeSlot: number;
  builds: PlayerBuild[];
  hoveredSkillId: string | null;

  setActiveSlot: (index: number) => void;
  renameBuild: (index: number, name: string) => void;
  unlockSkill: (skillId: string) => void;
  lockSkill: (skillId: string) => void;
  resetBuild: (index: number) => void;
  setHoveredSkill: (skillId: string | null) => void;
}
```

### Zod Schema (storage.ts)
```typescript
const DecisionEntrySchema = z.object({
  skillId: z.string(),
  towerLevel: z.number(),
  timestamp: z.number(),
});

const PlayerBuildSchema = z.object({
  slotIndex: z.number().min(0).max(4),
  name: z.string(),
  unlockedSkillIds: z.array(z.string()),
  decisionLog: z.array(DecisionEntrySchema),
  stats: z.object({
    damage: z.number(),
    movement: z.number(),
    reduction: z.number(),
  }),
  updatedAt: z.number(),
});
```

### Serialization Note
`PlayerBuild.unlockedSkillIds` is a `Set<string>` in memory but serialized as `string[]` for localStorage. The store's persist config needs `partialize` or custom `storage` to handle Set↔Array conversion.
