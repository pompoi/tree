# Task: Store migration and validation

## Description

Bump the localStorage storage key from `skill-tree-builds-v2` to `skill-tree-builds-v3` and update validation logic. Builds referencing removed skill IDs should gracefully reset to defaults.

## Acceptance Criteria

- [ ] Storage key changed from `skill-tree-builds-v2` to `skill-tree-builds-v3`
- [ ] `validateStoredState` correctly validates builds against the new set of 57 skill IDs
- [ ] Builds containing removed skill IDs (executioner, whirlwind, mirror-guard, fortress, phantom-step, overrun) fail validation and reset to defaults
- [ ] Builds containing only valid skill IDs load correctly
- [ ] `BASE_SKILL_IDS` still works (unchanged T0 base skills)
- [ ] Default build starts with only base skills unlocked
- [ ] Stats computation works with new skill statBonus values
- [ ] Fresh load with no localStorage creates valid default builds

## Technical Details

### Files affected
- `src/stores/build-store.ts` — Change `STORAGE_KEY` constant
- `src/lib/storage.ts` — Update `validateStoredState` to check against new valid skill IDs

### Implementation approach

**build-store.ts:**
```typescript
const STORAGE_KEY = "skill-tree-builds-v3"; // was v2
```

**storage.ts:**
The existing `validateStoredState` should already validate skill IDs against `SKILL_MAP`. Since `SKILL_MAP` is derived from the `SKILLS` array (updated in Task 001), validation will automatically reject builds with removed skill IDs — as long as it checks that all `unlockedSkillIds` exist in `SKILL_MAP`.

Verify this behavior:
1. Read `storage.ts` to understand current validation logic
2. Ensure it checks `unlockedSkillIds` against `SKILL_MAP`
3. If not, add that check
4. Test: a build with `["melee-attack", "executioner"]` should fail validation

### Edge case: T2 prerequisite changes
A build might have `["melee-attack", "power-hit", "lunge"]` — valid in v2 (lunge only needed power-hit) but invalid in v3 (lunge needs power-hit + quick-jab). The skill IDs are still valid, so validation passes, but the unlock state is inconsistent.

Decision: **Allow this.** The build panel shows the skills as unlocked. The player can continue building from there. The `canRemove` check prevents removing power-hit while lunge is unlocked. This is a graceful migration — no data loss.

Alternative: Reset any build that doesn't satisfy prerequisite rules. But this is destructive and unnecessary.
